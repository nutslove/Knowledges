# Webアプリのレンダリング方式（SPA / MPA / CSR / SSR / SSG / ISR）

> 「どこで HTML を組み立てるか」「いつ組み立てるか」で方式が分かれる。
> Vite / Next.js のどちらを選ぶか（→ [ディレクトリ構造.md](./ディレクトリ構造.md)）は、この方式の違いが背景にある。

---

## まず用語の整理

| 用語 | 正式名称 | 一言で |
|---|---|---|
| **CSR** | Client-Side Rendering | **ブラウザ側**でJSがHTMLを組み立てる |
| **SSR** | Server-Side Rendering | **サーバー側**でリクエストごとにHTMLを組み立てる |
| **SSG** | Static Site Generation | **ビルド時**にHTMLを全部作っておく（静的） |
| **ISR** | Incremental Static Regeneration | SSGを「一定時間ごとに再生成」する（Next.js独自） |
| **SPA** | Single Page Application | ページ遷移でHTMLを再取得しない**アプリの形態** |
| **MPA** | Multi Page Application | ページ遷移ごとにHTMLを取りに行く**従来型の形態** |

> ポイント：**SPA/MPA は「アプリの形態」**、**CSR/SSR/SSG/ISR は「HTMLの作り方」**。
> レイヤーが違うので、「SPA かつ CSR」「SSR だけど SPA的な操作感」のように組み合わさる。

---

## 1. SPA（Single Page Application）とは

**最初に1枚のHTMLだけを読み込み、以降の画面切り替えはJavaScriptがブラウザ内で行う**アプリ。
ページ遷移してもサーバーに新しいHTMLを取りに行かない（＝リロードが起きない）。

```
[初回] ブラウザ ──HTML(ほぼ空)+JS一式── サーバー
[以降] 画面遷移は JS が担当。データだけ API から取得（HTMLは取りに行かない）
       ブラウザ ──「データちょうだい(API)」── サーバー
              ←──── JSON ────
```

- **中身の描画は基本 CSR**（ブラウザでJSが組み立てる）。
- 画面遷移が速く、アプリっぽい操作感（Gmail, Notion, 管理画面など）。
- **Vite で React アプリを作ると、標準はこれ（SPA + CSR）**。

### メリット / デメリット

| メリット | デメリット |
|---|---|
| 遷移が速く操作感が良い | 初回表示が遅い（JS読み込み→実行してから描画） |
| サーバー負荷が軽い（APIだけ） | **SEOに弱い**（初期HTMLが空なのでクローラが中身を読めない場合がある） |
| フロント/バックを分離しやすい | 初回のJSバンドルが大きくなりがち |

---

## 2. MPA（Multi Page Application）とは

**ページ遷移のたびに、サーバーから完成したHTMLを取得する**従来型のWeb。
昔ながらの PHP サイトや、多くの企業サイト・ブログがこれ。

```
/home  にアクセス → サーバーが home.html を返す
/about に遷移      → サーバーが about.html を返す（都度リロード）
```

- ページごとに完全なHTMLが返るので **SEOに強い**。
- 遷移のたびに白い画面→再描画が入る（SPAより“もっさり”感）。

---

## 3. CSR / SSR / SSG / ISR（HTMLの作り方）

同じReactでも「HTMLをいつ・どこで作るか」で変わる。

### CSR（Client-Side Rendering）
```
サーバー: 中身が空のHTML + JS を返すだけ
ブラウザ: JSを実行して初めて画面が描かれる
```
- **Vite の標準**。SPA と相性が良い。
- 初期HTMLが空 → **SEO・初回表示に弱い**。

### SSR（Server-Side Rendering）
```
リクエストのたびに、サーバーがReactを実行して
「完成したHTML」を作って返す
```
- 初回から中身の入ったHTMLが返る → **SEOに強い・初回表示が速い**。
- リクエストごとにサーバーが処理 → サーバー負荷・応答時間はCSRより重い。
- **Next.js が得意とする領域**。

### SSG（Static Site Generation）
```
ビルド時（デプロイ前）に全ページのHTMLを作っておく
アクセス時はできあがったHTMLを返すだけ
```
- 一番速い・安い（ただの静的ファイル配信）。
- **内容が頻繁に変わらないサイト向け**（ブログ、ドキュメント、LP）。
- 内容更新のたびにビルドし直しが必要。

### ISR（Incremental Static Regeneration）※Next.js独自
```
SSG（静的）だけど、「◯秒ごとに裏で作り直す」設定ができる
```
- SSGの速さ＋「たまに更新される」を両立。
- 例：ニュース一覧を60秒ごとに再生成。

---

## 4. 一覧で比較

| 方式 | HTMLを作る場所/タイミング | SEO | 初回表示 | 向くもの |
|---|---|---|---|---|
| **CSR (SPA)** | ブラウザ / アクセス後 | △弱い | 遅め | 管理画面・社内ツール・アプリ |
| **SSR** | サーバー / リクエスト毎 | ◎強い | 速い | 動的でSEOも要る公開サイト（ECなど） |
| **SSG** | ビルド時 / 事前 | ◎強い | 最速 | ブログ・ドキュメント・LP |
| **ISR** | ビルド時＋定期再生成 | ◎強い | 最速 | たまに更新される公開サイト |

---

## 5. 結局どう選ぶか（Vite vs Next.js に戻る）

```
Q. SEO が重要？ or サーバー側でHTMLを作りたい？
   No  → SPA + CSR で十分     → Vite
         （管理画面・社内ツール・API を叩くだけのフロント）
   Yes → SSR / SSG / ISR が要る → Next.js
         （公開サイト・ブログ・EC・SEO重視）
```

- **フロント単体で、バックは別（API を叩くだけ）** → **Vite（SPA + CSR）**。
- **検索エンジンに載せたい公開サイト / サーバー処理も1つにまとめたい** → **Next.js**。
- Next.js はページ単位で CSR / SSR / SSG / ISR を**混在**できる（全部入りに近い）。

> 補足：「SEOに弱い」と言っても、Google は近年 JS を実行してクロールできるので
> CSRでも完全にダメではない。ただし SSR/SSG の方が確実で速い、という位置づけ。

---

## 6. 用語ミニ辞典

- **ハイドレーション (Hydration)**：SSR/SSGで返した「静的なHTML」に、ブラウザ側で
  JSを後付けして操作可能にする処理。「サーバー製の骨格に、JSで魂を入れる」イメージ。
- **バンドル (Bundle)**：多数のJS/CSSファイルを1つ（数個）にまとめたもの。Vite等が生成。
- **ビルド (Build)**：TypeScript/JSX を、ブラウザが読める形に変換・最適化する処理。

---

## まとめ

- **SPA/MPA = アプリの形態**、**CSR/SSR/SSG/ISR = HTMLの作り方**。層が違う。
- React + **Vite = SPA + CSR** が標準。フロント単体・SEO不要ならこれで十分。
- **SEOや初回表示、サーバー描画が要るなら Next.js**（SSR/SSG/ISRを使い分けられる）。
