# JavaScript (TypeScript / React) ディレクトリ構造のベストプラクティス

> **前提**：このドキュメントの大部分は基本的に **フロントエンド単体** を対象とした話。
> バックエンドとフロントエンドを分離する場合の全体像は末尾の
> 「[バックエンド/フロントエンド分離の場合](#バックエンドフロントエンド分離の場合)」を参照。

## 大原則

1. **機能（feature）単位でまとめる > 種類（type）単位でまとめる**
   小規模なら種類別（`components/`, `hooks/`, `utils/`）でもよいが、規模が大きくなると「1つの機能を直すのに複数フォルダを行き来する」問題が起きる。**Feature-based（Colocation）** が現代の主流。
2. **関連するものは近くに置く（Colocation）**
   コンポーネント・そのテスト・スタイル・hooks・型を同じフォルダにまとめる。「一緒に変更されるものは一緒に置く」。
3. **依存の方向を一方向にする**
   `features` → `shared`（共通）への依存はOK。逆（共通が個別機能に依存）はNG。`feature A` → `feature B` の横依存も避ける。
4. **公開APIを `index.ts` で絞る（barrel file）**
   各 feature / module はフォルダ直下の `index.ts` から「外に見せたいものだけ」export する。内部実装は外から触らせない。

## 推奨構造（中〜大規模 SPA / Feature-based）

```
src/
├── app/                  # アプリ全体の土台
│   ├── router/           # ルーティング定義
│   ├── providers/        # Context / Provider のまとめ (Theme, Query, Auth...)
│   ├── store/            # グローバル状態 (Redux / Zustand など)
│   └── App.tsx
│
├── pages/                # ルーティングの単位（ページ）。featureを組み合わせるだけ
│   ├── HomePage/
│   └── UserProfilePage/
│
├── features/             # ★ビジネスロジックを持つ機能単位
│   ├── auth/
│   │   ├── components/   # この機能専用のコンポーネント
│   │   ├── hooks/        # useAuth など
│   │   ├── api/          # API呼び出し
│   │   ├── types.ts
│   │   └── index.ts      # 公開API（外にはここ経由でアクセス）
│   └── cart/
│       └── ...
│
├── shared/ (or components/, lib/)  # 機能横断で再利用する“汎用”部品
│   ├── ui/               # Button, Modal, Input などドメイン非依存のUI
│   ├── hooks/            # 汎用hooks (useDebounce, useLocalStorage)
│   ├── lib/              # 外部ライブラリのラッパー (axios設定など)
│   └── utils/            # 純粋関数 (formatDate, ...)
│
├── types/                # グローバルな型定義 (.d.ts など)
├── assets/               # 画像・フォント
├── styles/               # グローバルCSS / テーマ
└── main.tsx              # エントリーポイント
```

### レイヤーの依存ルール
```
pages → features → shared
            ↑
        app (providers, store)
```
- `shared` は誰からも参照されるが、何も参照しない（最下層）。
- `features` 同士の直接依存は禁止。必要なら `pages` で組み合わせる。

## 小規模プロジェクトの場合（種類別でOK）

機能が少ないうちは過剰設計になるので、シンプルに：

```
src/
├── components/   # 再利用UI
├── pages/
├── hooks/
├── utils/
├── api/
├── types/
└── App.tsx
```
> 成長してきたら `features/` を導入してリファクタする、で十分。最初から完璧を狙わない。

## コンポーネント単位の Colocation 例

Button/

├── Button.module.css     # or styles.ts (CSS-in-JS)
├── Button.stories.tsx    # Storybook

## Next.js (App Router) の場合

`app/` ディレクトリがルーティング予約語なので、構造が少し変わる：

```
src/
├── app/              # ルーティング専用（page.tsx, layout.tsx, route.ts）
│   ├── (marketing)/  # Route Group（URLに影響しないグルーピング）
│   └── dashboard/
│       ├── page.tsx
│       └── _components/  # _始まりはルーティング対象外（コロケーション用）
├── features/
├── components/
├── lib/
└── hooks/
```
- `app/` 配下にはページ固有のものだけ。再利用ロジックは `app/` の外（`features/`, `components/`）へ。
- `_components` のように `_` プレフィックスを使うとルーティングから除外されコロケーションできる。

## 命名規則（参考）

| 対象 | 規則 | 例 |
|------|------|-----|
| コンポーネントファイル | PascalCase | `UserCard.tsx` |
| hooks | camelCase + `use` | `useAuth.ts` |
| ユーティリティ/その他 | camelCase | `formatDate.ts` |
| 定数 | UPPER_SNAKE_CASE | `MAX_COUNT` |
| 型 / interface | PascalCase | `type User`, `interface Props` |

> フォルダ名は kebab-case（`user-profile/`）か PascalCase か、チームで統一すれば良い。重要なのは一貫性。

## TypeScript Tips

- **パスエイリアス**を設定して相対パス地獄（`../../../`）を避ける。
  ```jsonc
  // tsconfig.json
  {
    "compilerOptions": {
      "baseUrl": "src",
      "paths": { "@/*": ["*"] }
    }
  }
  ```
  → `import { Button } from '@/shared/ui/Button'`
  （Vite/Next.js 側にも同じエイリアス設定が必要）

- 型は使う場所の近くに置く（`types.ts` をfeature内に）。本当にグローバルなものだけ `src/types/`。

- **barrel file（index.ts）は使いすぎ注意**：循環参照やビルド時のツリーシェイク阻害の原因になることがある。featureの「公開境界」としてだけ使い、内部で多用しない。

---

## アンチパターン

- ❌ 巨大な `components/` に全コンポーネントを平置き
- ❌ `utils/index.ts` に何でも詰め込む “神ファイル”
- ❌ feature 同士の相互 import（循環依存）
- ❌ 深すぎるネスト（`a/b/c/d/e/Component.tsx`）。2〜3階層を目安に
- ❌ 最初から大規模設計を入れて小規模なのに複雑化

---

## まとめ

> **「規模に応じて選ぶ」**
> - 小規模 → 種類別（type-based）でシンプルに
> - 中〜大規模 → 機能別（feature-based）＋ shared レイヤー＋一方向依存
> - 迷ったら「一緒に変わるものは一緒に置く（Colocation）」を基準に判断する

参考アーキテクチャ：[Feature-Sliced Design (FSD)](https://feature-sliced.design/) / [bulletproof-react](https://github.com/alan2207/bulletproof-react)

## バックエンド/フロントエンド分離の場合

これまでの `src/` 構成は **フロントエンド単体** を対象にしたもの。
バック/フロントを分離する場合、全体構成は2パターンある。

### パターン1: リポジトリを完全に分ける（シンプルで王道）

```
my-frontend/        ← これまでの src/ 構成がそのまま当てはまる
└── src/
    ├── app/
    ├── pages/
    ├── features/
    ├── shared/
    └── ...

my-backend/         ← 別リポジトリ（Go, Node, Python...）
└── ...
```

- フロントの `src/features/xxx/api/` は、バックエンドのREST/GraphQLを **HTTPで叩くだけ**。
- 型の共有は「OpenAPI/Swaggerから型生成」「protobuf」「共有npmパッケージ」などで行う。
- デプロイが独立していて簡単。フロント/バックでチームが別な場合に向く。

### パターン2: モノレポ（1リポジトリ内で分離）

```
my-app/
├── apps/
│   ├── frontend/    ← フロント（これまでの src/ 構成）
│   └── backend/     ← バックエンド
├── packages/        ← 共有コード
│   ├── types/       ← ★API型をフロント/バック両方で共有
│   ├── ui/
│   └── config/
├── package.json     ← workspaces (pnpm / turborepo など)
└── turbo.json
```

> `apps/frontend`・`apps/backend` は命名の一例。`web`・`api` などでもよく、
> チーム内で一貫していれば自由。

- **メリット**：型を `packages/types` で共有でき、フロント/バックの型ズレを防げる。
- ツール：pnpm workspaces + Turborepo / Nx が定番。
- 同じチーム・少人数で、型安全性を重視するなら◎。

### どちらを選ぶか

| | パターン1: リポジトリ分離 | パターン2: モノレポ |
|---|---|---|
| チーム | フロント/バックでチームが別 | 同じチーム・少人数 |
| 型共有 | OpenAPI型生成など一手間 | `packages/`で楽に共有 |
| デプロイ | 独立して簡単 | 設定がやや複雑 |
| おすすめ度 | シンプルで王道 | 型安全性を重視するなら◎ |

> **判断の目安**：まずは **パターン1（リポジトリ分離）** が素直。
> 将来 型のズレが気になってきたら、OpenAPIからの型自動生成、もしくはモノレポ化を検討する流れがおすすめ。
