# FastAPIのエラーハンドリングについて

FastAPIでは、エラー時に**適切なHTTPステータスコードとJSONボディ**をクライアントへ返すための仕組みが用意されている。中心となるのが `HTTPException` と、独自の変換ロジックを差し込む**例外ハンドラ**。

> 出典: [FastAPI公式 - Handling Errors](https://fastapi.tiangolo.com/tutorial/handling-errors/)

---

## 1. `HTTPException`（基本）

エンドポイントや依存関数の中で `raise` すると、FastAPIがそれを捕捉して**指定したステータスコードのJSONレスポンス**に変換する。

```python
from fastapi import FastAPI, HTTPException

app = FastAPI()

items = {"foo": "The Foo Item"}

@app.get("/items/{item_id}")
async def read_item(item_id: str):
    if item_id not in items:
        raise HTTPException(status_code=404, detail="Item not found")
    return {"item": items[item_id]}
```

- `raise` した時点で**以降の処理は実行されず**、レスポンスが返る（`return`と同じく処理を打ち切る）
- レスポンスボディは自動的に `{"detail": "Item not found"}` になる
- `detail` は文字列だけでなく **dict / list など任意のJSON化可能な値**を渡せる

```python
raise HTTPException(
    status_code=404,
    detail={"code": "ITEM_NOT_FOUND", "message": "指定の商品は存在しません"},
)
# → {"detail": {"code": "ITEM_NOT_FOUND", "message": "..."}}
```

### カスタムヘッダーを付ける

認証エラーで `WWW-Authenticate` を返すなど、ヘッダーを添えたい場合は `headers` 引数を使う。

```python
raise HTTPException(
    status_code=401,
    detail="Not authenticated",
    headers={"WWW-Authenticate": "Bearer"},
)
```

> [!NOTE]
> `HTTPException` は FastAPI が用意した**通常のPython例外**にすぎない。`return` の代わりに `raise` するだけで、FastAPIが例外ハンドラを通してHTTPレスポンスに変換してくれる。依存関数（`Depends`）の中で `raise` しても同じように機能するため、認証ガードなどで多用される（→[APIRouterについて](APIRouterについて.md) の `Depends`）。

## 2. バリデーションエラー（`422 Unprocessable Entity`）

Pydanticモデルやパス／クエリパラメータの型が合わない場合、FastAPIは**自動的に `422` を返す**。これは自分で `raise` しなくても発生する。

```json
// price に文字列を送った場合などのレスポンス例
{
  "detail": [
    {
      "type": "float_parsing",
      "loc": ["body", "price"],
      "msg": "Input should be a valid number, unable to parse string as a number",
      "input": "abc"
    }
  ]
}
```

- `loc` … エラーが起きた場所（`body` / `query` / `path` とフィールド名）
- `msg` … 人間向けメッセージ
- `type` … エラー種別

この自動バリデーションこそがPydanticモデルを使う最大の利点（→[request, responseのスキーマについて](request,%20responseのスキーマについて.md)）。

## 3. 例外ハンドラ（`@app.exception_handler`）

独自の例外クラスを定義し、それが送出されたときのレスポンスを**一箇所で**組み立てられる。ビジネスロジック層は「例外を投げるだけ」にでき、HTTP変換を分離できる。

```python
from fastapi import FastAPI, Request
from fastapi.responses import JSONResponse

app = FastAPI()


# 独自の例外（HTTPException を継承しなくてよい）
class UnicornException(Exception):
    def __init__(self, name: str):
        self.name = name


@app.exception_handler(UnicornException)
async def unicorn_exception_handler(request: Request, exc: UnicornException):
    return JSONResponse(
        status_code=418,
        content={"message": f"Oops! {exc.name} did something."},
    )


@app.get("/unicorns/{name}")
async def read_unicorn(name: str):
    if name == "yolo":
        raise UnicornException(name=name)   # ← ハンドラが捕捉して418を返す
    return {"unicorn_name": name}
```

- `@app.exception_handler(例外クラス)` で「この例外が投げられたらこう返す」を登録
- ハンドラは `(request, exc)` を受け取り、`Response`（`JSONResponse`等）を返す
- **ドメイン例外 → HTTPレスポンス** の変換を集約でき、各エンドポイントが綺麗になる

## 4. デフォルトハンドラの上書き

FastAPI標準の `RequestValidationError`（422）や `HTTPException` の**レスポンス形式そのものを差し替える**こともできる。APIのエラー形式を全社共通フォーマットに揃えたいときに使う。

```python
from fastapi import FastAPI, Request, status
from fastapi.exceptions import RequestValidationError
from fastapi.responses import JSONResponse
from fastapi.encoders import jsonable_encoder


@app.exception_handler(RequestValidationError)
async def validation_exception_handler(request: Request, exc: RequestValidationError):
    return JSONResponse(
        status_code=status.HTTP_422_UNPROCESSABLE_ENTITY,
        content=jsonable_encoder({
            "error": "VALIDATION_ERROR",
            "details": exc.errors(),   # 元のエラー詳細（list）
            "body": exc.body,          # 受け取った不正な入力そのもの
        }),
    )
```

- `exc.errors()` で元のバリデーションエラー内容、`exc.body` で受信した不正な入力を取り出せる
- `jsonable_encoder` を通すと、`errors()` に含まれる非JSON型（例外オブジェクト等）も安全にシリアライズできる

> [!WARNING]
> `HTTPException` のレスポンス形式を全体で差し替えたい場合、対象にするのは **FastAPI の `HTTPException` ではなく Starlette の `HTTPException`**。FastAPIが内部で送出するものも含めて捕捉するため。
>
> ```python
> from starlette.exceptions import HTTPException as StarletteHTTPException
>
> @app.exception_handler(StarletteHTTPException)
> async def http_exception_handler(request, exc):
>     return JSONResponse(
>         status_code=exc.status_code,
>         content={"error": exc.detail},
>     )
> ```

### 既定のハンドラを再利用する

独自ハンドラの中でログだけ足して、変換自体はFastAPI標準に委ねることもできる。

```python
from fastapi.exception_handlers import (
    http_exception_handler,
    request_validation_exception_handler,
)
from starlette.exceptions import HTTPException as StarletteHTTPException


@app.exception_handler(StarletteHTTPException)
async def custom_http_exception_handler(request, exc):
    print(f"HTTPエラー発生: {repr(exc)}")           # ログだけ追加
    return await http_exception_handler(request, exc)  # 変換は標準に委譲
```

> [!TIP]
> `status` モジュール（`from fastapi import status`）を使うと、`status.HTTP_404_NOT_FOUND` のように**定数で**ステータスコードを書ける。数値の打ち間違いを防げてエディタ補完も効く。

## 5. どれを使うか（使い分け）

| やりたいこと | 手段 |
|---|---|
| 単発で「404を返す」など | `raise HTTPException(...)` |
| 入力の型・必須チェック | 何もしない（Pydanticが自動で422） |
| ドメイン例外をHTTPに変換したい | `@app.exception_handler(独自例外)` |
| エラーレスポンスの形式を全体で統一したい | デフォルトハンドラ（`RequestValidationError` / Starlette `HTTPException`）を上書き |

---

## ポイントまとめ

- 単発のエラーは `raise HTTPException(status_code=..., detail=...)` が基本。`detail` はdict等も可、`headers` も付けられる
- 型・必須のバリデーションエラー（422）は**自動**で返るので自分で書かない。詳細は `exc.errors()` / `exc.body`
- ドメイン層の例外は独自クラス＋`@app.exception_handler` でHTTP変換を集約すると綺麗
- エラー形式を統一したいなら `RequestValidationError` / **Starletteの** `HTTPException` のハンドラを上書きする
- 依存関数（`Depends`）内で `raise` しても同じく機能する（認証ガードで多用）
