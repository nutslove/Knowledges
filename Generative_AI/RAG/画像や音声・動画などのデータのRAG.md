# 画像や音声・動画などのデータのRAG（マルチモーダルRAG）

## 前提：Embeddingモデルは「保存」しないし、中身を「解析（言語化）」もしない
- マルチモーダルEmbeddingモデル（CLIPやその派生など）は、**入力（テキスト/画像/音声/動画）をベクトルに変換する推論器**
  - ファイルを「保存」する機能は持っていない → 原本の保存先は**アプリ側の設計次第**
- 「解析」という言葉には2つのレイヤーがあるので注意
  - **①特徴抽出→ベクトル化**：Embeddingモデルも、入力を内部で処理して**意味的な特徴を抽出＝ある種の「理解・解析」をした上でベクトルに変換**している（この意味では解析している）
  - **②中身の言語化**：ただしその出力は **類似検索のための数値表現（ベクトル）** であって、**人間やLLMが読んで推論・回答に使えるコンテンツ（説明文・ラベル・抽出情報）ではない**
  - ＝Embeddingは①はするが②はしない。「意味的に近いものを探す」ためのベクトルは作れるが、**画像の中身を言語化・詳細に説明する（VLM的な処理）はしない**
- つまり、マルチモーダルRAGは「**検索用ベクトルを作るモデル**」と「**中身を言語化して理解・回答するモデル（VLM）**」で役割分担する構成になる

---

## 原本ファイルはどこに格納するか

### テキストベースRAGの場合
- チャンク化したテキストをEmbedding
- ベクトルDB（pgvector, Pinecone, Weaviateなど）に「**ベクトル + チャンクのテキスト本文**（payload/metadata）」を一緒に格納
- テキストチャンクは軽量なので、DBのペイロードに直接入れても問題になりにくい

### マルチモーダル（画像・音声・動画）の場合
- 画像・動画・音声は**容量が大きく**、ベクトルDBは大容量バイナリの格納・配信に最適化されていない
- そのため一般的に**原本とベクトルを分離**する
  - **原本ファイル自体** → オブジェクトストレージ（S3, GCS, Azure Blob など）や既存のファイルサーバーなどに格納
  - **ベクトルDB側** → embeddingベクトル + **メタデータ（原本へのURL/パス、file_id、サムネイルURLなどの参照情報）のみ**を格納
- 検索時は、ベクトルDBで類似度検索 → ヒットしたレコードのメタデータから、オブジェクトストレージ上の原本を取得する**2段構成**

> [!NOTE]
> テキストのように「チャンク＋本文をDBに同居」させるのではなく、
> 「**原本はストレージ、DBはベクトル+参照情報のみ**」という分離アーキテクチャがマルチモーダルの基本。

### 補足
- **画像**：軽量なサムネイル（縮小版）だけメタデータに埋め込んでプレビュー用に使うケースもある
- **動画・音声**：通常フレーム/セグメント単位でchunk的に区切ってembeddingし、各chunkが「元動画のtimestamp範囲」を指すメタデータを持つ設計が多い
  - テキストのchunk化と考え方は同じ。ただし「chunk=テキスト片」ではなく「**chunk=時間区間や画像パッチ**」になるイメージ

## 検索でヒットした原本をどう「解析」するか
- Embeddingは検索までしか担当しない → 取得した画像・動画を実際に解析するには**別のモデル/処理を挟む**

### 典型的な後段処理のパターン
1. **VLM（Vision-Language Model）に投げる**
   - 取得した画像/動画フレームを、Claude・GPT-4o・Geminiのようなマルチモーダル対応LLMに渡す
   - VLMに渡すプロンプト次第で、用途は次の2通りになる（＝「解析」と「回答生成」は同じVLMがやる）
     - **単に中身を説明させる**：「この画像に写っているものを説明して」「このシーンで起きていることを教えて」など。画像や動画などのデータに対する説明文（テキスト）を得たいとき
     - **そのまま最終回答まで出させる**：`画像 ＋ ユーザーの実際の質問` を一緒に渡すと、VLMが画像を見た上で**1回の呼び出しで最終回答を生成**する（中間の説明文を挟まない）
   - つまりRAGの最終回答生成も、多くはこのVLMがそのまま担う（別のLLMを後段に置かない）
   - ※VLMを「説明文の生成」だけに使い、その説明文を別のテキストLLMに渡して回答させる2段構成もある → [ブリッジパターン](#agent本体がマルチモーダル非対応の場合ブリッジパターン)を参照
2. **専用の解析モデルを使う**（特定タスクに特化しているのでVLMより精度・速度で有利な場合がある）
   - 物体検出・セグメンテーション（YOLO, SAMなど）
   - OCR（画像内テキスト抽出）
   - 動画なら行動認識・シーン分割、音声トラックがあればWhisperで文字起こし
3. **動画はフレームサンプリング**（動画全体をそのままVLMに渡すのは非効率・高コスト）
   - キーフレームを抽出（シーンチェンジ検出など）して各フレームを画像として解析
   - あるいは動画対応モデル（Gemini系など）に直接動画ファイルを渡す

## Agent（LLM）本体もマルチモーダル対応が必要
- テキストのように「文字列としてプロンプトに埋め込む」ことは画像・動画では**できない**
- Agent本体のLLMが実際に画像/動画を扱って回答するには、**そのLLM自身がマルチモーダル対応**である必要がある

### どうやってLLMに渡すか
- マルチモーダル対応LLMのAPIでは、メッセージの`content`に**テキストブロックと画像ブロックを並べて渡す**形式になっている
- 「プロンプト文字列に埋め込む」のではなく、**モデルへの入力（トークン列）としてテキストトークンと画像トークンを混在させて渡す**
- 画像はモデル内部の**vision encoder**でトークン化され、テキストと同じTransformerに一緒に流し込まれる
  - これがマルチモーダルLLMの本質的な仕組み
  - 「非対応モデル」はそもそもこの画像トークン化の仕組みを持っていないので画像を理解できない

Anthropic（Claude）APIの例：
```javascript
messages: [
  {
    role: "user",
    content: [
      {
        type: "image",
        source: { type: "base64", media_type: "image/jpeg", data: imageBase64Data }
      },
      { type: "text", text: "この画像に写っているものを説明してください" }
    ]
  }
]
```

### 動画の場合
- そのまま渡せるモデル（Gemini系など）もあるが、多くの場合は次のように変換してから渡す
  - **キーフレームを複数枚抽出**して画像として並べて渡す
  - 音声トラックがあれば別途文字起こし（Whisperなど）して**テキストとして添付**
- ＝「画像の連続 ＋ テキスト」に変換して渡す

### Agent本体がマルチモーダル非対応の場合（ブリッジパターン）
- テキスト専用Agentでも、間接的に画像を扱えるようにできる
  1. 別途VLM（マルチモーダルモデル）に画像/動画を渡し、内容を**テキストで説明させる**（キャプション生成、OCR結果など）
  2. そのテキスト化された結果を、テキストのみのAgentのプロンプトに埋め込む
- ＝「マルチモーダルモデルをtool/functionとして呼び出し、その出力（テキスト）をAgentに渡す」構成
- Agent自体がマルチモーダルでなくても、「**画像解析ツール**」を持たせることで擬似的に画像を扱える

### まとめ
| Agent本体のLLM | 渡し方 | 特徴 |
| --- | --- | --- |
| マルチモーダル対応 | 画像/動画をトークンとして直接入力に混ぜて渡す | 一番自然で精度も高い |
| テキスト専用 | 別のVLMをツールとして呼び、テキスト化した結果を渡す | 間接的だが実装は柔軟 |

- どちらを取るかは、使っているAgentフレームワーク（LangChain, LangGraph, 独自実装など）と**コスト・レイテンシ要件**次第

## PDFなど図・表を含むドキュメントの場合
- PDFのRAGでは普通、`PyPDFLoader` などのDocumentLoaderで**テキスト化 → 原文チャンク＋ベクトルをDBに格納**する
- ただしDocumentLoaderは**テキストレイヤーだけ**を抽出するので、**PDF内の図・グラフ・複雑な表・スキャン画像は落ちる**（中身がスキップされる／崩れる）
  - ＝そのままEmbeddingしても、図の情報はベクトルにもDBにも入らない
- 対応方法は大きく3系統

### ① 図・表を抽出して「テキスト化」してから通常のテキストRAGに載せる（最も一般的）
- PDFを**要素単位（テキスト/図/表）に分解**するパーサーを使う
  - `unstructured`, `LlamaParse`, Azure Document Intelligence, AWS Textract, Google Document AI など
- 図・グラフは**VLMでキャプション（説明文）生成**、表は**Markdown/HTMLテーブルに構造化**
- ポイントは、**図・グラフ・表があった場所を、その「テキスト（説明文）」に置き換えてしまう**こと
  - 以降そのテキスト（説明文）は、**元から普通のテキストだったかのように**扱われる
  - つまり、**この置き換わったテキスト（元は図・グラフ・表だったもの）が、検索用のベクトル（Embedding）にも、回答用の原文テキストにも、両方入る**（通常のテキストチャンクと全く同じ流れ）
- 実務では、テキスト（説明文）と一緒に**元の図画像へのパス/URLもメタデータに持たせておく**とよい
  - 回答時に「根拠はこの図です」と**元画像を提示**できる
  - 説明文だけで足りないときは、**VLMに元画像を渡して**さらに詳しく見させられる（②や本文のVLM解析につながる）
- ＝[ブリッジパターン](#agent本体がマルチモーダル非対応の場合ブリッジパターン)を、PDFの前処理段階でやるイメージ

### ② 図を「画像のまま」マルチモーダルEmbeddingして別系統で検索
- 基本の流れ
  1. **先に分解**：PDFをテキストと、図・グラフ・表に分ける
  2. **原本は別々に保存**：テキストチャンクはそのまま、図・グラフは**画像としてObject Storage**に保存
  3. **ベクトル化**：テキストと画像をそれぞれベクトル化する（「別空間」か「同一空間」かは下記のどちらでもよい）
  4. **回答時**：それぞれから検索・取得し、**マージしてから回答**する
     - この「マージ」の中身（テキストチャンクと図の画像をどう1つの回答入力にまとめるか）は、**回答モデルがVLMかテキスト専用か**で変わる：
       - **回答モデルがVLM（マルチモーダル対応）→ テキスト変換は不要**
         - テキストチャンクと図の画像を、**そのままVLMの入力（content）に一緒に並べて渡す**だけ
         - ＝[`content`にテキストブロックと画像ブロックを並べる](#どうやってllmに渡すか)形式。この場合の「マージ」は**1つのプロンプトに同梱する**という意味（画像をテキストに変換して混ぜるのではない）
       - **回答モデルがテキスト専用 → 画像を先にテキスト化してからマージ**
         - まず**画像をVLMでテキスト化（キャプション/OCR）** → テキスト同士でマージして回答
         - ＝[ブリッジパターン](#agent本体がマルチモーダル非対応の場合ブリッジパターン)
- ①との違いは、図を**テキストに置き換えず「画像のまま」検索対象にする**こと。ヒットした図の画像はそのまま**VLMに渡して回答**する
- ＝このノート本体で説明している「画像RAG」の構成を、PDFの図に適用したもの

#### ベクトルを「別空間」にするか「同一空間」にするか
実装上の差はここだけで、**マージが検索の中で自然に起きるか／検索後に自分でやるか**の違い。

- **同一空間（共有ベクトル空間）**
  - テキストも画像も **同じマルチモーダルEmbeddingモデル（CLIP系）** で、同じ空間にベクトル化
  - 1回のクエリ検索で**テキストと画像を混ぜて一気に取得**できる → マージが自然に済む
- **別空間**
  - テキストは**テキスト用Embedding**、画像は**画像用（CLIP系）Embedding**と、別モデル・別インデックス
  - クエリを**それぞれの系統で別々に検索** → 結果を**後でマージ**する
  - この場合、系統ごとにスコアの尺度が違うので、**スコアの正規化やRerank**が必要になることがある

### ③ ページを丸ごと画像化して検索する（新しめのアプローチ）
- テキスト抽出を**一切せず**、PDFの各ページを**画像1枚として扱う**
- **ColPali / ColQwen** のような「ドキュメント画像用の検索モデル」でページ画像を直接Embedding → 検索
- ヒットしたページ画像をそのままVLMに渡して回答
- レイアウト・図・表を含めてページの見た目ごと扱えるので、**パーサーで構造化する手間が要らない**のが利点

### どれを選ぶか
| 状況 | 選択 |
| --- | --- |
| 図が補足程度／既存のテキストRAG資産を使いたい | ①（実務で最も一般的） |
| 図・グラフそのものを検索対象にしたい（例：類似チャートを探す） | ② |
| レイアウトが複雑・スキャンPDF中心・パース精度に悩んでいる | ③ |

## 全体のパイプラインイメージ
```
ユーザークエリ（テキスト or 画像）
   ↓ embedding化
ベクトルDBで類似検索
   ↓ ヒットしたmetadataからfile_id/URL取得
Object Storageから原本ファイル取得
   ↓
VLM/専用解析モデルに原本を渡して解析
（Agent本体がマルチモーダル対応なら、原本を直接contentに混ぜて渡す）
   ↓
解析結果（説明文・抽出情報）をユーザーへの回答に統合
```
- テキストRAGの「検索したチャンクをLLMのコンテキストに入れて回答生成」と同じ発想で、
  画像/動画も「**取得したものをVLMのコンテキストに入れて回答生成**」させる形になる
